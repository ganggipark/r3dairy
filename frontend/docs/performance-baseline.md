> **추정 근거**:
>
> `assembleDailyContent`는 19개 generate 함수를 순차 호출하며,
> 각 함수는 O(1) 룩업 테이블 조회 및 문자열 연결만 수행합니다 (반복문 없음).
>
> `analyzeDailyFortune`은 60간지 모듈러 산술(O(1)) 및 십성 맵 조회(O(1))만 수행합니다.
>
> `calculateSaju`는 동일 프로필에 대해 LRU 캐시(최대 200항목)가 적용되어
> 첫 호출 이후 365일 반복 시 캐시 히트로 원국 계산 비용은 1회만 발생합니다.
>
> V8 JIT 최적화를 고려하면 365회 반복도 단순 산술+문자열 연결 수준이므로
> 500ms를 초과할 가능성이 낮습니다.

### DOM 노드 추정 (diary-print/page.tsx 기준)

| 구분 | 페이지 수 | 예상 DOM 노드 수 |
|------|----------|----------------|
| 1일  | 4페이지   | ~220개          |
| 30일 | 62페이지  | ~6,020개        |
| 365일| 732페이지 | ~72,620개       |

**추정 근거**:

```
Left page  (1일) : 10개 블록 × 평균 12 DOM 노드 = 약 120 노드
Right page (1일) : 시간 그리드(19행) + 할 일(4행) + 감사(3행) + 메모(8행) + 헤더 등 = 약 80 노드
커버 + 빈 페이지 : 약 20 노드
1일 합계          : 120 + 80 + 20(커버/빈) = 약 220 노드

365일 합계        : 20 + 365 × (120 + 80) = 73,020 노드
```

> **참고**: React는 `display: none`인 노드도 DOM에 유지합니다.
> preview 모드에서 현재 페이지만 보여주더라도 732페이지 전체 DOM이 존재합니다.

## 현재 렌더링 구조 분석 (diary-print/page.tsx)

### 렌더링 방식

- `days.map(...)` 으로 모든 날짜 페이지를 **한 번에 DOM에 생성** (가상화/윈도잉 없음)
- preview 모드: `display: none` CSS로 비활성 페이지 숨김 — DOM은 모두 유지됨
- scroll 모드: 전체 페이지 동시 렌더링

### 현재 최적화 현황

| 항목 | 상태 |
|------|------|
| 사주 원국 계산 LRU 캐시 (200항목) | ✅ 적용 |
| API 청크 로딩 (31일 단위, 프로그레스바) | ✅ 적용 |
| React 컴포넌트 가상화 | ❌ 미적용 |
| 콘텐츠 윈도잉 (visible range만 DOM 유지) | ❌ 미적용 |
| 페이지 lazy render (IntersectionObserver) | ❌ 미적용 |

### 성능 위험 요소

1. **DOM 규모**: 365일 시 약 73,000 DOM 노드 — React 최초 렌더 및 scroll 이벤트에 부담
2. **preview 모드의 display:none**: 노드가 DOM에 남아 메모리를 점유하지만 paint/layout 비용은 절약됨
3. **콘텐츠 데이터 로딩**: API에서 365일치 DailyContentResponse 배열을 `days` state에 보유 — JSON 크기에 따라 수십 MB 가능

## 권장 사항

현재 **콘텐츠 생성 시간 및 메모리는 모두 임계값 이내**입니다.
추가 최적화는 **불필요**합니다.

단, 다음 상황 발생 시 재평가가 필요합니다:

- 실제 브라우저에서 365일 scroll 모드 FPS가 30 이하로 떨어질 때
  → `react-virtual` 또는 커스텀 IntersectionObserver 윈도잉 구현
- `days` state의 메모리가 200MB를 초과할 때 (API 응답 크기 증가 시)
  → 현재 보이는 ±10일 범위만 state에 유지하는 슬라이딩 윈도우 구현
- 30일 API 응답이 1초를 초과할 때
  → 백엔드 API 레벨 최적화 (배치 캐싱, Supabase 인덱스 검토)
