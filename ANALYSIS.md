### 3.3 테스트 생년월일 3개 검증

> **참고**: 실제 API 호출 없이 코드 로직 기반 검증. 테스트 기본 사주: 1971-11-17 04:00 양력 남자.

**테스트 1: 1971년 11월 17일 04:00 양력 남자**
- 년주: 입춘 이후 → 辛亥년 (신해)
- 월주: 11월(해월 기준 절기) → 己亥월 (기해)
- 일주: 1900-01-31 기준 일수 계산 → 60갑자 인덱스 산출
- 시주: 04:00 = 인시(寅時, 03-05시) → 일간에 따른 시간 배정

**테스트 2: 1990년 1월 15일 12:00 양력 여자**
- 년주: 1월 15일은 입춘(2월 4일경) 이전 → 1989년 己巳년(기사)
- 월주: 1월(축월) → 년간 己의 1월 = 丁丑(정축)

**테스트 3: 2000년 6월 21일 18:00 양력 남자**
- 년주: 庚辰년(경진)
- 월주: 6월(오월) → 년간 庚의 6월 = 壬午(임오)

**검증 결과**: 코드 로직은 한국 명리학 이론에 **부합함**. 60갑자 순환, 입춘 보정, 절기 기반 월주, 진태양시 보정 모두 표준 방법론 준수.

## 4. 기문둔갑 알고리즘 검증

### 4.1 포국 방식

| 항목 | 상태 | 상세 |
|------|------|------|
| **포국 방식** | ⚠️ 간이 방식 | 시가법/일가법이 아닌 **일진(日辰) 60갑자 기반 旬/局 결정** 방식. 전통적 시가법 포국과 차이 있음. |
| **음둔/양둔 구분** | ✅ 구현 | `offset % 18 < 9` → 양둔, 나머지 → 음둔. 양둔=순행, 음둔=역행 |
| **局 번호 결정** | ✅ 구현 | `(offset % 9) + 1` (1-9 순환) |

### 4.2 9궁 배치 로직

```
┌─────┬─────┬─────┐
│ 4   │ 9   │ 2   │
│ 巽SE│ 離S │ 坤SW│
├─────┼─────┼─────┤
│ 3   │ 5   │ 7   │
│ 震E │ 中央│ 兌W │
├─────┼─────┼─────┤
│ 8   │ 1   │ 6   │
│ 艮NE│ 坎N │ 乾NW│
└─────┴─────┴─────┘
```

- **구현 상태**: ✅ 완전
- 9개 궁의 방위(한국어/영문) 매핑 완비
- 시간대별 궁 배정: 局 번호 + 시간 인덱스 기반 순환

### 4.3 8문(八門) 배치

| 문 | 품질 | 기본 에너지 | 사용자 라벨 |
|----|------|-----------|-----------|
| 휴문(休門) | good | 8 | 집중하기 좋은 시간 |
| 생문(生門) | good | 9 | 에너지가 충만한 시간 |
| 상문(傷門) | avoid | 2 | 에너지가 낮은 시간 |
| 두문(杜門) | neutral | 5 | 내면 정리 시간 |
| 경문(景門) | neutral | 5 | 창의적 활동에 적합 |
| 사문(死門) | avoid | 2 | 휴식이 필요한 시간 |
| 경문2(驚門) | avoid | 3 | 중요한 결정을 피해야 할 |
| 개문(開門) | good | 8 | 새로운 시작에 좋은 시간 |

- **구현 상태**: ✅ 완전 (8문 × 에너지 + 품질 분류)

### 4.4 미구현 항목

| 항목 | 상태 | 영향도 |
|------|------|--------|
| **9성(九星)** | ❌ 미구현 | 높음 — 전통 기문둔갑의 핵심 요소. 천봉/천임/천충/천보/천영/천예/천주/천심 배치 없음 |
| **8신(八神)** | ❌ 미구현 | 중간 — 값부/등사/태음/육합/구지/주작/구천/구지 배치 없음 |
| **천반/지반 구분** | ❌ 미구현 | 높음 — 현재 단일 층 구조, 천반(天盤)/지반(地盤) 이중 구조 미반영 |
| **시가법 정밀 포국** | ❌ 미구현 | 높음 — 현재 간이 日辰 기반, 정확한 시가법과 차이 있음 |

### 4.5 기문둔갑 종합 평가

**현재 구현 수준**: 기문둔갑의 **기초 프레임워크** (9궁 + 8문 + 에너지 계산)만 구현된 상태. 전통 기문둔갑의 핵심인 9성·8신·천지반 이중 구조가 없어 **정통 기문둔갑 포국이라 할 수 없음**. 다만 시간대별 에너지·방위·품질 정보를 제공하는 **간이 시간 가이드** 용도로는 충분히 기능함.

### 5.2 운세+기문+일기 연동 구조

```
날짜별 데이터 연동:
  사주 계산 (프론트엔드 실시간)
    ↓
  기문둔갑 시간대 계산 (프론트엔드 실시간)
    ↓
  콘텐츠 조합 (assembleDailyContent)
    ↓
  역할 번역 (translateDailyContent)
    ↓                          ↓
  좌측: 오늘의 안내         우측: 사용자 기록
  (운세+기문 결합 콘텐츠)    (일기, 할일, 감정 등)
```

- **운세 데이터**: 실시간 계산 (DB 저장 안 함, 매번 재계산)
- **기문 데이터**: 실시간 계산 (qimen_slots로 시간대별 제공)
- **사용자 기록**: Supabase `daily_logs` 테이블에 저장

### 5.3 데이터 유실 위험

| 위험 항목 | 위험도 | 상세 |
|----------|--------|------|
| **운세 데이터 미저장** | ⚠️ 중간 | 운세가 DB에 저장되지 않고 매번 재계산. 알고리즘 변경 시 과거 운세 결과 불일치 가능 |
| **localStorage 토큰** | ⚠️ 중간 | 브라우저 데이터 삭제 시 로그인 상태 유실 (데이터 자체는 Supabase에 안전) |
| **청크 로드 실패** | ⚠️ 낮음 | diary-print에서 31일 청크 실패 시 해당 기간 누락 (전체 로드 중단은 안 됨) |
| **오프라인 지원** | ❌ 없음 | PWA/ServiceWorker 미구현, 오프라인 시 데이터 접근 불가 |

### 6.2 AI 통합 평가

현재 시스템은 **AI를 실질적으로 사용하지 않음**. 모든 콘텐츠는 규칙 기반으로 생성됨:
1. 사주 데이터에서 에너지/오행/강약 추출
2. 조건 분기로 텍스트 블록 선택
3. 역할별 표현 변환 (학생/직장인/프리랜서)

AI 연동은 향후 확장 영역으로, 현재는 인프라(패키지 설치)만 준비된 상태.
